# Правила для контрибуторов MDK2

Спасибо за интерес к проекту! Прежде чем вносить изменения, пожалуйста, ознакомьтесь с этим документом.



## С чего начать

1. Сделай форк репозитория
2. Клонируй его локально:
```bash
git clone https://github.com/your-username/mdk2.git
cd mdk2
```
3. Создай ветку для своих изменений:
```bash
git checkout -b feat/название-фичи
```



## Ветки

| Ветка | Назначение |
|-------|-----------|
| `main` | Продакшн. Прямые коммиты запрещены |
| `dev` | Основная ветка разработки |
| `feat/...` | Новая функциональность |
| `fix/...` | Исправление багов |
| `docs/...` | Изменения документации |

Все изменения вносятся через Pull Request в ветку `dev`.



## Коммиты

Используй понятные сообщения на русском или английском. Формат:

```
feat: добавить фильтрацию меню по категориям
fix: исправить ошибку 500 при загрузке меню
docs: обновить README
refactor: вынести логику корзины в отдельный модуль
style: поправить отступы в cartUI.js
```

Префиксы:
- `feat` — новая функциональность
- `fix` — исправление бага
- `docs` — документация
- `refactor` — рефакторинг без изменения поведения
- `style` — форматирование, отступы
- `chore` — рутинные задачи (зависимости, конфиги)



## Код

### Frontend (JavaScript)
- Используй ES модули (`import/export`)
- Не добавляй сторонние библиотеки без обсуждения
- Не хардкодь URL — используй `config.js`
- Проверяй наличие элементов перед работой с DOM (`if (!element) return`)

### Backend (Go)
- Следуй стандартному стилю Go (`gofmt`)
- Логируй все значимые операции в формате `[LOG]`, `[WARN]`, `[ERROR]`
- Новые эндпоинты добавляй в `main.go` и документируй в README
- Не храни секреты в коде — только через переменные окружения
- Обрабатывай все ошибки явно, не игнорируй `err`


## Pull Request

1. Убедись что код работает локально
2. Опиши что изменил и зачем
3. Если это баг — опиши как воспроизвести
4. Прикрепи скриншот если изменения визуальные
5. Один PR — одна задача

Шаблон описания PR:
```
## Что сделано
- ...

## Зачем
- ...

## Как проверить
- ...
```


## Что не принимается

- Коммиты напрямую в `main`
- Хардкод секретов, паролей, токенов
- Сторонние зависимости без обсуждения с командой
- Изменения в чужой зоне ответственности без согласования
- Код без обработки ошибок
